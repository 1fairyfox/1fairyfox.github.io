## Update-command layer — nodes can check the system for updates (0.4.9)

Continuation of the lifecycle-runbook work. Fairy Fox wanted each project to behave like a
node that knows how to check the system for updates — say "check system updates" (or anything
like it) and the AI knows to get the newest version and pull the relevant file — with a level
of hub integration that's real but won't break things unasked.

Decided (FF's call) on **check-and-report, then wait**: the safest autonomy level — discover
and explain, change nothing without a clear go-ahead. Also corrected course mid-task: FF noted
this shouldn't be a rigid keyword/function-call list — AI is natural language — so it's framed
as an *intent to recognize in plain conversation*, not magic phrases.

Changes:

- `hub/standards/adopting-updates.md` — new opening section "Checking for updates — what to do
  when asked": recognize the update intent naturally, run refresh → diff → **report → stop**,
  apply only on an explicit go-ahead. Plus the "why bounded this way" rationale tying it to the
  anti-recursion rules.
- `hub/templates/CLAUDE.md` — expanded the cross-project section into a standing instruction so
  every adopting node inherits the check-for-updates behavior and its guardrails (on-request,
  read-only, git-ignored clone, reconcile-not-clobber, no apply/history-rewrite without go-ahead).
- `CLAUDE.md` (this repo) — added the hub-side reading: the hub never adopts from itself, so
  "check for updates" here = scan siblings for round-ups + internal consistency, same
  report-first posture.
